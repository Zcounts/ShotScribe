# ShotScribe Image Storage Autopsy and Plan

## 1. Executive Summary
- The repo currently uses **Convex File Storage** for cloud storyboard images (`ctx.storage.generateUploadUrl`, `ctx.storage.getUrl`, `ctx.storage.delete`) with metadata in `projectAssets`. This is visible in `convex/assets.ts`, `convex/schema.ts`, `src/services/assetService.ts`, and `src/components/ShotCard.jsx`.
- Corrected product direction is now:
  1. **S3 bucket must be private** (no public-read bucket/object policy).
  2. **Reads must use signed/private delivery**, not public bucket URLs.
  3. **Not a per-shot gallery**. We need a **project-level media library**.
  4. Shots should keep a **single selected storyboard image reference** (unless current code proves otherwise).
- Recommended implementation shape (minimal-risk, compatible with current app):
  - Store media assets as project-level records in Convex (library entries).
  - Store shot assignment separately (shot -> selected asset reference).
  - Upload binary directly from browser to private S3 using presigned upload intent.
  - Resolve image display via signed/private delivery generated by trusted backend path.
  - Use soft-delete + undo + delayed hard-delete (24h) with Convex `runAfter` and recurring reconciliation cron.
- Backward compatibility must preserve existing shot data that currently contains `shot.image`, `shot.imageAsset`, and `shot.imageAsset.cloud.assetId` style references while we migrate to the project media-library model.

## 2. Current State in the Repo (Anchor Points)

### 2.1 Frontend behavior today
- `src/components/ShotCard.jsx`
  - `handleImageChange` uploads image for cloud projects and then writes shot image state.
  - Current UX is effectively **upload-per-shot** and immediate assignment.
- `src/services/assetService.ts`
  - Upload path currently asks Convex for upload URL and writes binary to Convex storage.
  - Returns asset payload used to hydrate shot image fields.
- `src/store.js`
  - Shot-level image fields persisted in project data include `shot.image` + `shot.imageAsset` (cloud/local variants).

### 2.2 Backend behavior today
- `convex/schema.ts`
  - `projectAssets` includes `shotId` and Convex storage IDs (`thumbStorageId`, `fullStorageId`).
- `convex/assets.ts`
  - `createAssetUploadUrl`, `completeAssetUpload`, `getAssetView`, `pruneOrphanedAssets` are centered on Convex storage IDs.
- Current shape is metadata+binary both coupled to Convex storage semantics, not S3 object keys.

### 2.3 Gap vs corrected requirements
- Current model is too shot-centric and storage-provider-specific.
- Correct requirement needs project library first, shot assignment second.
- Current delete semantics are immediate prune patterns tied to save flow; corrected requirement needs **two-stage delete with undo and 24h delayed hard delete**.

## 3. Recommended Architecture (Corrected)

### 3.1 Core model split (required)
1. **Project Media Library Assets**
   - Canonical record for uploaded images in a project.
   - Lifecycle + metadata + storage key live here.
2. **Shot-to-Asset Assignment**
   - A shot stores selected image reference (single chosen storyboard image).
   - Unassigning from shot does **not** delete asset by default.

This explicit separation is required to support “choose from existing library” and safe deletion behavior.

### 3.2 Storage and delivery
- S3 bucket remains **private**.
- Upload: browser -> private S3 via presigned URL from Convex action.
- Read: private delivery only (short-lived signed URL or equivalent private CDN signing), issued via trusted backend pathway.
- No public bucket URL flow in app code.

### 3.3 UI behavior (required flows)

#### A) Upload new image (from storyboard “Add Image”)
1. User clicks **Add Image** on shot.
2. Modal/sheet opens with two choices: **Upload New** or **Choose from Library**.
3. If Upload New:
   - client normalizes image (existing pipeline can remain initially),
   - upload intent requested,
   - binary uploaded to S3,
   - asset finalized in project media library,
   - newly created asset auto-assigned to current shot.

#### B) Choose from library (from storyboard “Add Image”)
1. User clicks **Add Image**.
2. Opens project media library picker.
3. User selects an existing asset.
4. Shot assignment updates to selected asset (single selected image).

#### C) Remove image from shot
1. User clicks **Remove from Shot** (or equivalent).
2. App unlinks shot from asset.
3. Asset remains in project media library unless explicitly deleted there.

#### D) Delete from media library
1. User deletes asset from library.
2. Asset is immediately soft-deleted/hidden from default library views.
3. Undo affordance appears (toast/banner + library “Recently deleted” state).
4. Any active shot assignment should be cleared or blocked according to policy (recommended: clear assignment references at soft-delete boundary to avoid rendering deleted media).
5. Schedule hard-delete check for +24h.

#### E) Undo delete
1. User triggers undo within allowed window.
2. Soft-delete flag clears; asset reappears in library.
3. Hard-delete job detects restored state and no-ops.

### 3.4 Delayed-delete lifecycle (required)

**State machine**
- `active` -> `soft_deleted` -> (`restored` OR `hard_deleted`).

**Mechanics**
- On soft delete:
  - set `deletedAt`, `deleteStatus='soft_deleted'`, `hardDeleteAfter = now + 24h`.
  - call `runAfter` for targeted hard-delete attempt at `hardDeleteAfter`.
- At hard-delete attempt:
  - if asset restored (`deletedAt` cleared): stop.
  - if still referenced by any shot: stop and mark `deleteStatus='blocked_referenced'`.
  - if unreferenced: delete S3 object, then mark `hard_deleted` (or remove row if desired, but tombstone is safer).
- Safety net:
  - recurring Convex cron scans due assets and retries failed/blocked candidates.
  - cron also catches missed `runAfter` executions and transient S3 failures.

### 3.5 Backward compatibility strategy
- Continue reading legacy shot image structures (`shot.image`, `shot.imageAsset`, existing `cloud.assetId`) while introducing new library-first records.
- Migration path should avoid forcing immediate destructive rewrites:
  - treat existing shot-linked cloud assets as initial library assets,
  - keep one selected image per shot,
  - preserve local project behavior unchanged.

## 4. Proposed Data Model

> Keep this minimal and compatible. Names can be adapted to Convex conventions, but separation must remain explicit.

### 4.1 `projectMediaAssets` (library records)
- `projectId`
- `uploadedByUserId`
- `kind` (`storyboard_image`)
- `provider` (`s3`)
- `bucket` (optional if implicit from env)
- `objectKey`
- `mime`
- `fileSizeBytes`
- `width`
- `height`
- `sourceName`
- `createdAt`
- `updatedAt`
- `deletedAt` (nullable)
- `hardDeleteAfter` (nullable)
- `deleteStatus` (`active` | `soft_deleted` | `blocked_referenced` | `hard_deleted` | `delete_failed`)

Suggested indexes:
- by project + active status for library listing
- by project + deleteStatus + hardDeleteAfter for reconciliation

### 4.2 Shot assignment (separate)
**Preferred minimal approach for current repo:**
- Keep shot payload with a single selected asset ID reference (`imageAsset.cloud.assetId` compatible path), while progressively converging field naming.
- Do not introduce multi-image shot gallery model unless a hard requirement appears.

Optional normalized table only if needed later:
- `shotAssetAssignments` with `(projectId, shotId, assetId, assignedAt)` and unique `(projectId, shotId)`.

### 4.3 Compatibility with current schema
- Existing `projectAssets` can be evolved instead of replaced immediately, but must stop encoding core semantics as per-shot gallery data.
- Existing `shotId` on asset rows should become optional/legacy and not the primary ownership model.

- Keep existing shot rendering compatible with legacy and new references.

- Update storyboard Add Image entry point to present:
  - Upload New
  - Choose from Library

- Implement remove-from-shot unlink behavior.

## Phase 3 — Two-stage delete lifecycle
- Library delete performs soft-delete + immediate hide + undo support.
- Schedule hard-delete with `runAfter` for 24h.
- Hard-delete executes only when still unreferenced.
- Add recurring reconciliation cron as safety net for missed jobs/failures.

## Phase 4 — Legacy data compatibility and cleanup
- Ensure legacy single-image shot data remains readable/editable.
- Backfill/normalize where low-risk.
- Remove obsolete code paths only after validation.

## 7. Testing and Verification Plan

### 7.1 Upload and assignment
- Upload New from storyboard:
  - creates media library asset,
  - assigns to shot,
  - image renders after reload.
- Choose from Library:
  - no new upload,
  - assignment swaps to selected asset.

### 7.2 Unlink vs delete semantics
- Remove from shot:
  - shot clears selected image,
  - asset still visible in media library.
- Delete from library:
  - asset hidden immediately,
  - undo restores visibility.

### 7.3 Delayed hard-delete safety
- Soft-delete sets `hardDeleteAfter = now + 24h` and schedules `runAfter`.
- If restored before deadline, hard-delete no-ops.
- If referenced at deadline, object is not deleted and status reflects blocked state.
- Reconciliation cron catches due items and retries failures.

### 7.4 Security and privacy
- Confirm bucket/object are not publicly readable.
- Confirm rendered image delivery uses signed/private URLs only.
- Confirm unauthorized user cannot access asset delivery endpoint/URL generation path.

### 7.5 Backward compatibility
- Existing projects with legacy shot image fields still render.
- Legacy cloud asset references can still resolve or gracefully migrate.
- No regression to local-only project image behavior.

## 8. Open Questions
1. Should we evolve existing `projectAssets` in-place or introduce `projectMediaAssets` + migration shim?
2. For soft-deleted assets that are currently assigned to shots, do we clear shot assignment immediately or keep placeholder until user chooses replacement? (recommended: clear assignment to avoid hidden/deleted render ambiguity)
3. What is the undo UX window expectation in UI (entire 24h vs short toast action + restore from “Recently deleted”)?
4. Signed delivery mechanism choice for launch: direct short-lived signed S3 URLs vs private CDN signed URLs?
5. Do we retain tombstone rows indefinitely after hard delete for audit/debug, or compact after retention window?
6. Should mobile library UX ship in same phase as desktop storyboard picker, or immediately after desktop stabilization?
